Designing a distributed lock is one of the most critical and heavily scrutinized challenges in distributed systems. A poorly designed lock can lead to data corruption (if mutual exclusion fails) or system halts (if deadlocks occur). 

Let's break down the Low-Level Design (LLD) for a robust distributed lock system. For this design, we will use a **Redis-based approach** (similar to the Redlock algorithm) as it is the industry standard for low-latency, high-throughput distributed locking, though we will discuss the trade-offs of this choice.

---

### 1. API Contracts

The contract must be simple but enforce safety mechanisms like Time-To-Live (TTL) to prevent deadlocks if a client crashes.

```java
interface DistributedLock {
    /**
     * Attempts to acquire a lock on a resource.
     * @param resourceId The unique identifier for the resource.
     * @param clientId The unique identifier for the client/thread requesting the lock.
     * @param ttlMillis Time to live in milliseconds before auto-release.
     * @return true if acquired, false otherwise.
     */
    boolean acquire(String resourceId, String clientId, long ttlMillis);

    /**
     * Releases the lock, ensuring the client only releases its own lock.
     * @param resourceId The unique identifier for the resource.
     * @param clientId The unique identifier for the client.
     * @return true if successfully released, false if it didn't hold the lock.
     */
    boolean release(String resourceId, String clientId);

    /**
     * Extends the lease of an existing lock (Heartbeat mechanism).
     * @param resourceId The unique identifier for the resource.
     * @param clientId The unique identifier for the client.
     * @param additionalTtlMillis Time to add to the lease.
     * @return true if renewed, false if the lock was lost.
     */
    boolean renew(String resourceId, String clientId, long additionalTtlMillis);
}
```

---

### 2. Database / Storage Schema

We will use a Key-Value store (Redis). We don't need a traditional relational schema; instead, we need a specific atomic data structure.

* **Key:** `lock:{resourceId}` (e.g., `lock:order_12345`)
* **Value:** `clientId` (A UUID generated by the client instance + thread ID).
* **Metadata:** TTL (Time-To-Live) managed natively by the KV store.

**Why this structure?**
The value *must* be the `clientId`. If a client just sets a generic "LOCKED" value, a delayed client might accidentally release a lock that has already expired and been acquired by a completely different client.

### 3. Core Classes & Data Structures

```java
public class RedisDistributedLock implements DistributedLock {
    private final RedisClient redisClient;

    public RedisDistributedLock(RedisClient redisClient) {
        this.redisClient = redisClient;
    }

    @Override
    public boolean acquire(String resourceId, String clientId, long ttlMillis) {
        // SET resourceId clientId NX PX ttlMillis
        // NX = Only set if it does Not eXist.
        // PX = Set expiration in milliseconds.
        String response = redisClient.set(resourceId, clientId, "NX", "PX", ttlMillis);
        return "OK".equals(response);
    }

    @Override
    public boolean release(String resourceId, String clientId) {
        // CRITICAL: We cannot just do GET then DEL. 
        // We must use a Lua script for atomicity.
        String luaScript = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "   return redis.call('del', KEYS[1]) " +
            "else " +
            "   return 0 " +
            "end";
        
        Long result = (Long) redisClient.eval(luaScript, Collections.singletonList(resourceId), Collections.singletonList(clientId));
        return result != null && result == 1L;
    }

    @Override
    public boolean renew(String resourceId, String clientId, long additionalTtlMillis) {
        // Also requires a Lua script for atomicity.
        String luaScript = 
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "   return redis.call('pexpire', KEYS[1], ARGV[2]) " +
            "else " +
            "   return 0 " +
            "end";
        Long result = (Long) redisClient.eval(luaScript, Collections.singletonList(resourceId), Arrays.asList(clientId, String.valueOf(additionalTtlMillis)));
        return result != null && result == 1L;
    }
}
```

### 4. Concurrency & Edge Cases (The "Gotchas")

Handling concurrency in a distributed lock is where the real engineering happens. 

* **The "Delete Someone Else's Lock" Problem:** * *Scenario:* Client A acquires a lock. Client A experiences a long Garbage Collection (GC) pause. The lock's TTL expires. Client B acquires the lock. Client A wakes up and calls `release()`.
    * *Solution:* This is why the `clientId` is stored as the value, and the `release` method uses a **Lua Script** to check the ID before deleting. Client A will fail to delete Client B's lock because the `clientId` no longer matches.
* **The "Task Takes Too Long" Problem:**
    * *Scenario:* A client acquires a lock with a 10s TTL, but the database transaction it's performing takes 15s. The lock expires at 10s, another client jumps in, and mutual exclusion is violated.
    * *Solution:* **Watchdogs (Heartbeats)**. A background daemon thread in the client periodically wakes up (e.g., every 3 seconds) and calls `renew()` to extend the TTL as long as the primary thread is still executing. If the client actually crashes, the watchdog dies too, and the lock naturally expires.
* **The Split-Brain / GC Pause Execution Problem:**
    * *Scenario:* Even with Lua scripts, if Client A gets paused by GC, the lock expires, Client B gets the lock, and writes to the DB. Client A wakes up, thinks it still holds the lock, and *also* writes to the DB.
    * *Solution:* **Fencing Tokens**. The lock manager should return a monotonically increasing integer (e.g., lock iteration #45). When the client writes to the underlying database, it passes this token. The database must reject any token less than the highest one it has seen.

### 5. System Trade-offs

| Feature | Redis (AP System) | ZooKeeper / etcd (CP System) |
| :--- | :--- | :--- |
| **Performance** | **High** (In-memory, minimal latency). | **Moderate** (Requires consensus protocols like Raft/ZAB). |
| **Consistency** | **Weak/Eventual**. If the Redis master crashes before syncing to the replica, a lock can be acquired twice. | **Strong**. Data is replicated to a quorum before confirming the lock. |
| **Ideal Use Case**| Efficiency locks (preventing two workers from doing the same expensive calculation, where occasional failure is okay). | Correctness locks (preventing double-billing a user, where failure means corrupted financial data). |

*Note: The Redlock algorithm attempts to make Redis safer by requesting locks from multiple independent Redis nodes, but it still relies heavily on synchronized system clocks, which can be dangerous in distributed environments.*

### 6. Key Flows

**Happy Path:**
1. Client A generates UUID `client-A-123`.
2. Client A calls `acquire("resource-X", "client-A-123", 10000)`.
3. Redis executes `SET NX PX`, returns `OK`.
4. Client A performs its critical section.
5. Client A calls `release("resource-X", "client-A-123")`.
6. Redis executes Lua script, matches ID, deletes key.

**Failure Path (Client Crash):**
1. Client A acquires lock on "resource-X" for 10 seconds.
2. Client A's server abruptly loses power.
3. Client B tries to acquire lock, fails.
4. After 10 seconds, Redis automatically evicts the key due to TTL.
5. Client B retries and successfully acquires the lock. No deadlock occurs.
