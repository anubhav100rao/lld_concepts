## Study Strategy

**Pattern recognition order:**

1. Classic sorted array BS → Lower/upper bound → Rotated arrays
2. Binary search on answer (minimize max / maximize min) — **this is the #1 interview pattern**
3. Kth element via BS + counting function
4. BS on matrix → BS on monotonic/abstract functions
5. BS + greedy / sliding window / prefix sum combos
6. BS + DP (LIS patience sorting, egg drop, job scheduling)
7. BS + graph (threshold-based reachability)
8. BS on strings (rolling hash combos)
9. Design / online problems using BS
10. Floating point BS, ternary search

**Senior/Staff round priorities:**

- **BS on answer** is the single most tested pattern — master the template: define search space on the answer, write `canAchieve(mid)`, binary search the boundary. Problems like (410) Split Array, (1011) Ship Packages, (875) Koko, (1482) Bouquets are canonical
- **Kth smallest via BS + count**: (4) Median of Two Arrays, (719) Kth Pair Distance, (668) Kth in Multiplication Table — the trick is binary searching the answer and counting how many elements are ≤ mid
- **BS + graph reachability**: (1631) Path With Minimum Effort, (778) Swim in Rising Water — binary search the threshold, then BFS/DFS/UF to check feasibility
- **BS + DP optimization**: (300) LIS in O(n log n) via patience sorting, (887) Super Egg Drop monotonicity
- **Floating point BS**: know when to use `eps`-based termination vs fixed iteration count (typically 100 iterations suffices for double precision)
- **Rotated array family**: understand the invariant — which half is sorted — and handle duplicates gracefully

**The BS on answer template (memorize this):**

```
lo, hi = min_possible_answer, max_possible_answer
while lo < hi:
    mid = lo + (hi - lo) // 2
    if canAchieve(mid):
        hi = mid          # minimize answer
        # lo = mid + 1    # (for maximize: lo = mid)
    else:
        lo = mid + 1      # minimize answer
        # hi = mid - 1    # (for maximize: hi = mid)
return lo
```

**Common interview traps:**

- Off-by-one: `lo < hi` vs `lo <= hi` — know which to use and why
- Integer overflow in `(lo + hi) / 2` — always use `lo + (hi - lo) / 2`
- Floating point BS: don't use `lo < hi`, use iteration count or `hi - lo > eps`
- Forgetting to handle duplicates in rotated array (worst case O(n))
- Not identifying the monotonic property — if `f(x)` isn't monotonic, BS doesn't apply directly
- Search space definition: is it index-based or value-based? Getting this wrong leads to subtle bugs
