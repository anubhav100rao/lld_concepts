## Study Strategy

**Pattern mastery order:**
1. Sort + greedy (two pointer pairing, interval scheduling)
2. Interval merging / activity selection (the classic greedy family)
3. Jump game / range extension
4. Greedy array transformation (Kadane's, gas station, candy)
5. String construction (Remove K Digits, lexicographic greedy)
6. Greedy + heap (regret-based exchange)
7. Greedy + binary search (minimize-max, maximize-min)
8. Counting / frequency greedy (partition labels, task scheduler)
9. Two-pass greedy (candy, trapping rain water)
10. Exchange argument proofs (staff-level maturity)

**Senior/Staff round priorities:**

- **Exchange argument**: the gold standard for proving greedy correctness. The pattern: assume you have an optimal solution where two adjacent elements are "out of order" by your greedy criterion, show swapping them doesn't worsen the solution. (179) Largest Number, (857) Hire K Workers, (1029) Two City Scheduling, (630) Course Schedule III all use this
- **Regret-based greedy + heap**: greedily commit to a choice, but keep a max-heap of past decisions so you can "undo" the worst one when a better option appears. (630), (871), (1642) are canonical — this pattern is a staff-level differentiator
- **Greedy + binary search**: define feasibility via greedy simulation, binary search the answer. (410), (1011), (875), (2071). The greedy `canAchieve(mid)` function is the crux — getting this right is the interview
- **Candy problem (135)**: two-pass greedy is a pattern that generalizes — left-to-right enforces one constraint, right-to-left enforces another, take max. Also seen in (42) Trapping Rain Water
- **Interval scheduling**: (435) Non-overlapping Intervals is the purest form — sort by end time, greedily pick earliest-ending non-conflicting interval. Know why sorting by end (not start) is correct
- **Partition Labels (763)**: greedy with "last occurrence" tracking — extend partition boundary until you've seen all occurrences of included characters. Simple but tests understanding
- **Patching Array (330)**: a beautiful greedy problem that rarely appears but deeply tests greedy intuition — maintain a reachable range [1, miss), if miss ≤ current number add it, else "patch" by adding miss itself

**How to prove a greedy solution is correct:**

1. **Greedy stays ahead**: show that at every step, the greedy solution is at least as good as any other solution. Used for activity selection, interval scheduling
2. **Exchange argument**: show any optimal solution can be transformed into the greedy solution by swapping elements without worsening cost. Used for custom sorting (179, 1029, 857)
3. **Structural argument**: show the problem has optimal substructure + greedy choice property. Used for Huffman, MST
4. **Contradiction**: assume greedy is suboptimal, derive a contradiction. Often combined with exchange

**Common interview traps:**
- Greedy doesn't always work — the hardest part is knowing WHEN it applies. If you can't prove correctness, it might be DP
- Sorting criterion matters enormously: sort by end (activity selection) vs sort by deadline (course schedule) vs sort by ratio (hire workers) — wrong sort = wrong answer
- Two-pass greedy: forgetting to take max/min of both passes (candy problem)
- Exchange argument: proving for adjacent swaps is sufficient for global optimality only if the comparator is transitive
- Gas station: the "total gas ≥ total cost" check is necessary but not sufficient for arbitrary starting points — you need the prefix-sum argument
- Interval problems: open vs closed endpoints, overlapping at single point — edge cases that change answers
