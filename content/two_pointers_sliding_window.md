## Study Strategy

**Pattern mastery order:**

**Two Pointers track:**
1. Opposite direction — pair sum (167, 15, 16)
2. Opposite direction — container/area (11, 42)
3. Opposite direction — palindrome/reversal (125, 680)
4. Same direction — read/write filtering (26, 27, 283)
5. Same direction — merge two sequences (88, 986)
6. Fast-slow — cycle detection (141, 142, 287)
7. Multi-pointer / k-pointer (15, 75, 632)
8. Two pointers on strings + linked lists

**Sliding Window track:**
1. Fixed-size window (643, 438, 567, 239)
2. Variable — longest/maximum (3, 424, 1004)
3. Variable — shortest/minimum (209, 76, 727)
4. "Exactly K" via atMost trick (992, 1248, 930)
5. Window + hash map / freq counter
6. Window + monotonic deque (239, 1425, 862)
7. Window + heap / multiset (480, 1438)
8. Circular window (918, 2134, 1423)
9. 2D / matrix window (85, 1074, 363)
10. Shrinkable vs non-shrinkable templates

**Senior/Staff round priorities:**

- **"Exactly K" decomposition**: the atMost(K) − atMost(K−1) trick is the single most important advanced sliding window technique. (992) Subarrays with K Different Integers is the canonical problem. It transforms a hard counting problem into two standard sliding window problems. Know it cold
- **Shrinkable vs Non-shrinkable windows**: two distinct templates that are often confused:
  - **Shrinkable**: window shrinks aggressively to find minimum valid window. `while (condition) shrink left`. Used for "shortest" problems (76, 209)
  - **Non-shrinkable**: window never shrinks, only slides. `if (condition) move left once`. Used for "longest" problems where you want max window size (424, 1004). The window size is monotonically non-decreasing — this is an optimization over shrinkable
- **Sliding window + monotonic deque**: (239) Sliding Window Maximum is the gateway. The deque maintains decreasing order so front = max. Extends to DP optimization (1425, 1696) where dp[i] = max(dp[j]) + ... over a window. Staff-level pattern
- **Container With Most Water (11)**: classic "greedy two pointer" proof — always move the shorter wall because moving the taller wall can never increase area (width decreases, height capped by min). Interviewers test whether you can articulate WHY this is correct
- **Minimum Window Substring (76)**: the ultimate variable-size window problem. Expand right until valid, shrink left greedily, track minimum. Requires a "formed" counter tracking when all characters are satisfied — managing this counter cleanly is the hard part
- **Count Subarrays With Fixed Bounds (2444)**: a newer pattern — track positions of last min, last max, and last out-of-range. Count valid subarrays ending at each position. Tests compositional thinking
- **Fast-slow on arrays (287)**: Find the Duplicate Number using Floyd's on array indices as a graph — interviewers love this because it tests whether you can see the implicit graph. Phase 1: detect cycle. Phase 2: find entrance

**Templates:**

```python
# ==== SHRINKABLE (find minimum valid window) ====
def min_window(s, condition):
    left = 0
    result = float('inf')
    state = {}
    for right in range(len(s)):
        # expand: add s[right] to state
        while is_valid(state):
            result = min(result, right - left + 1)
            # shrink: remove s[left] from state
            left += 1
    return result

# ==== NON-SHRINKABLE (find maximum valid window) ====
def max_window(s, condition):
    left = 0
    state = {}
    for right in range(len(s)):
        # expand: add s[right] to state
        if not is_valid(state):
            # shrink: remove s[left] from state (only once!)
            left += 1
    return len(s) - left  # final window size = answer

# ==== EXACTLY K via atMost ====
def exactly_k(nums, k):
    return at_most(nums, k) - at_most(nums, k - 1)

def at_most(nums, k):
    left = 0
    count = 0
    state = {}
    for right in range(len(nums)):
        # expand: add nums[right], update distinct count
        while distinct(state) > k:
            # shrink
            left += 1
        count += right - left + 1  # all subarrays ending at right
    return count

# ==== TWO POINTER PAIR SUM ====
def two_sum_sorted(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo < hi:
        s = nums[lo] + nums[hi]
        if s == target:
            return [lo, hi]
        elif s < target:
            lo += 1
        else:
            hi -= 1
    return []

# ==== FAST-SLOW CYCLE DETECTION ====
def find_cycle_start(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:  # cycle detected
            slow = head   # phase 2: find entrance
            while slow != fast:
                slow = slow.next
                fast = fast.next
            return slow
    return None
```

**Common interview traps:**
- **Window state management**: forgetting to decrement hashmap counts when shrinking — leads to wrong "distinct" or "formed" counts
- **Off-by-one in counting**: `count += right - left + 1` counts all subarrays ENDING at right. Forgetting the +1 undercounts
- **Non-shrinkable pitfall**: using `while` instead of `if` makes it shrinkable — changes correctness for "longest" problems
- **Circular windows**: either double the array and take windows of original size, or use modular arithmetic. Don't try both
- **Two pointer validity**: pointers must move monotonically in one direction — if you need backtracking, it's not a two-pointer problem
- **3Sum duplicates**: skip duplicates for both the outer loop AND inner two-pointer loop, otherwise you get duplicate triplets
- **Fast-slow on linked lists**: not checking `fast.next` before `fast.next.next` → null pointer
- **Sliding window on negative numbers**: standard sliding window for "sum ≥ K" DOESN'T WORK with negatives — need monotonic deque on prefix sums (862)
